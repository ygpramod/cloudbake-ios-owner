# CloudKit Backup Operations

This runbook governs the private CloudKit schema used by CloudBake disaster-recovery backups. The
backup implementation remains dormant until the scheduling and Settings slices activate it.

## Container And Environments

- Team ID: `4H787CNDS2`
- Container: `iCloud.com.cloudbake.owner`
- Database: private database
- Custom zone: `CloudBakeBackup`

Develop and validate schema changes in the development environment. Promote the reviewed schema to
production before distributing a build that can write new record types or fields. Never use the
production environment to discover schema implicitly.

The app target must retain the CloudKit service and container entitlement. Xcode automatic signing
must produce a development or distribution profile containing the same container before a device or
archive build is considered valid.

## Private Schema

All records live in the `CloudBakeBackup` custom zone.

| Record type | Purpose | Fields |
| --- | --- | --- |
| `CBBackupPointer` | The sole published-generation pointer; record name `current` | `generationID` String, `updatedAt` Date/Time |
| `CBBackupGeneration` | Metadata and cleanup claim for one staged generation | `generationID` String, `createdAt` Date/Time, `formatVersion` Int64, `databaseSchemaVersion` String, `minimumCompatibleAppVersion` String, `payloadByteCount` Int64, `uploadByteCount` Int64, `fileCount` Int64, `lifecycleState` String |
| `CBBackupFile` | One immutable manifest, database, or asset payload | `generationID` String, `role` String, `relativePath` String, `byteCount` Int64, `sha256` String, `payload` Asset |

Make `CBBackupGeneration` queryable by record type and add a queryable index for
`CBBackupFile.generationID` in the development schema. No domain values,
customer names, order details, or original photo filenames belong in record names. Generation and
file record names are opaque identifiers generated by the app.

## Reproducible Schema Workflow

Authenticate `cktool` with the Apple developer account authorized for the team, then keep the
exported schema outside source control until it has been inspected for account- or environment-local
metadata.

```sh
xcrun cktool export-schema \
  --team-id 4H787CNDS2 \
  --container-id iCloud.com.cloudbake.owner \
  --environment development \
  --output-file /tmp/cloudbake-cloudkit-development.ckdb

xcrun cktool validate-schema \
  --team-id 4H787CNDS2 \
  --container-id iCloud.com.cloudbake.owner \
  --environment development \
  --file /tmp/cloudbake-cloudkit-development.ckdb
```

Before production promotion:

1. Compare the exported record types, field types, and queryability with this runbook and the exact
   app release.
2. Validate the development schema with `cktool`.
3. Promote the schema in CloudKit Console from development to production using the owning team.
4. Export and inspect the production schema as release evidence.
5. Perform the development-container smoke check below before release, and repeat against a signed
   pre-release build after production promotion.

Schema promotion is additive. Do not remove or change a deployed field type while a released build
can still read it. A future format change must update the backup compatibility contract first.

## Development-Container Smoke Check

This check intentionally runs outside CI because CI must not require a personal iCloud account.

1. Sign in to an iPhone with a development iCloud account and install a development-signed build
   whose profile contains `iCloud.com.cloudbake.owner`.
2. Seed a small local database plus one app-managed photo asset.
3. Capture and publish generation A; verify the pointer, manifest, database, and asset records in the
   private development database.
4. Capture generation B and interrupt each pre-pointer phase in turn. Confirm generation A remains
   current and restorable.
5. Retry generation B. Confirm publication succeeds, generation B becomes current, and abandoned or
   superseded generations are removed.
6. Disconnect networking during upload. Confirm the app reports a safe category and opaque operation
   ID, then succeeds on retry without duplicating the current generation.
7. Confirm no customer, order, recipe, inventory, supplier, note, tag, or original filename appears
   in CloudKit record names or diagnostics.
8. Record the build SHA, device, account environment, timestamps, and result in the release evidence.

The owner-facing backup controls and status are introduced by later slices; until then, invoke this
check only through a development harness or targeted integration tooling.

## Failure Safety

The current pointer is the publication boundary. Upload and verify every file in a non-current
generation before conditionally changing it. Cleanup first atomically claims a generation by moving
its lifecycle from `uploading` to `deleting` while proving that the unchanged pointer does not
reference it. Pointer publication also conditionally saves the unchanged generation, so it cannot
race past that claim. Existing file records are discovered by a paginated membership query and
deleted in bounded batches; every batch conditionally preserves the unchanged pointer. A cleanup
failure after publication is reported as pending cleanup and must never roll back or hide the newly
published generation.
